# Help Me Read —— 问题解决标准操作流程

> 🔴 **按需加载**：本文件仅处理 issue 时读取。正常学习流程不加载。

> 从发现问题到防止回归的六阶段标准化流程。每次解决 issue 时严格遵循，不得跳步。

---

## TL;DR 速查

| 阶段 | 做什么 | 产出 | 关键门禁 |
|------|--------|------|---------|
| **Phase 1** 发现与分诊 | 写入 ISSUES.md，定严重度，关联阶段码 | ISSUES.md 新条目 | 条目字段完整 |
| **Phase 2** 排序 · 方案 · 确认 | 排依赖顺序 → 逐条出方案 → **用户逐条确认** | 排序计划 + 确认后的方案集 | 🚧 全部确认前不动代码 |
| **Phase 3** 根因 + Advisor | 对照 failure-modes，复现，定位，**分派 Advisor 审查** | 根因描述 + Advisor 报告 | Advisor 审查通过 |
| **Phase 4** 集中修复 | 全部方案确认后，**一次性集中修改**所有文件 | 修改后的文件 | 不逐条改、不边确认边改 |
| **Phase 5** 测试执行 | 修复完成后**统一跑测试** + 人工检查 + 提交准备 | 测试报告 + commit message | 🚧 等用户确认 commit |
| **Phase 6** 文档 · 关闭 · 防回归 | 更新 ISSUES / failure-modes / README + 补测试覆盖 | 关闭的 issue + 零盲区 | 条款闭环 · 测试覆盖 |

> **三条铁律**：
> 1. **Phase 2 全部方案确认后才动手改代码**——不逐条改、不边确认边改
> 2. **Phase 5 改完统一跑测试**——不跳过、不分开跑
> 3. **Phase 5 提交前必须展示 changed files + test results + commit message，等用户确认**——不直接 commit

## Phase 1: 发现与分诊

### 1.1 问题来源

- 用户报告（对话中指出的异常行为）
- agent 自行检测（生成过程中发现的异常）
- 测试失败（TESTING.md T1-T13 任一模块未通过）
- 产物审查（手动检查输出文件时发现）

### 1.2 写入 ISSUES.md（标准化格式）

在 `ISSUES.md` 当前日期分组下添加新条目。**必须包含以下全部字段，不得省略：**

```markdown
#### <circled-number> <简短标题>

**现象**：<1-3 句：观察到了什么、何时发生、触发条件>

**复现步骤**（选填）：<其他人按此步骤操作能复现>

**问题分析**：<根因假设。哪个机制/环节失效了？>

**建议方向**：
- <修复方案 1>
- <修复方案 2>
- <推荐方案及理由>

**涉及文件**：<预估需要修改的文件列表>
**关联 failure-mode**：<已有 F-code 或"待新增">

---
```

> 编号规则：使用带圈数字（① ② … ㉑ ㉒ …），放在日期分组下。新条目取下一个可用编号。

### 1.3 严重度分类

| 级别 | 定义 | 响应策略 |
|------|------|----------|
| 🔴 **Blocker** | 管线无法完成（配置不可达、PDF 损坏、全部图片失败） | 立即修复，不等待批量 |
| 🟠 **Major** | 产物不正确/不完整（缺少导航、噪音概念、链接断裂） | 下一批次修复 |
| 🟡 **Minor** | 外观/优化/边界情况（多余日志、临时文件清理） | 累积后批量修复 |

### 1.4 关联 failure-modes 阶段码

根据症状所属的管线阶段，标注对应的 F-code。若 `references/failure-modes.md` 已有匹配条目，在 ISSUES.md 中引用；若无匹配，标记"待新增"。

| 阶段 | F-code 范围 |
|------|-------------|
| 步骤 0：配置 | F0-* |
| 步骤 1：获取与图片 | F1-* |
| 步骤 2：课程/笔记/概念生成 | F2-* |
| 步骤 3-4：学习阶段 | F3-* |
| 步骤 5：原子笔记填写 | F4-* |
| 产物后检查 | F5-* |

### Phase 1 出口

- [ ] ISSUES.md 条目包含全部必填字段（现象、问题分析、建议方向、涉及文件、关联 failure-mode）
- [ ] 严重度已判定，响应策略已明确
- [ ] F-code 已标注

## Phase 2: 排序 · 方案 · 确认

> ⚠️ **本阶段不动任何代码。** 全部方案经用户确认后，Phase 4 才动手。

### 2.1 按依赖排序

若同时解决多个 issue，先按以下规则排出修复顺序：

1. **基础设施先行**：配置文件/测试模式等底层问题先修（如 F0-*），上层产物问题后修（如 F5-*）
2. **共享文件集中**：多个 issue 改同一文件（如 SKILL.md）→ 合并到同一批次，避免反复修改冲突
3. **依赖链顺序**：A 的修复依赖 B 的修复结果 → B 先修

排序结果以表格呈现：

| 顺序 | Issue | 涉及文件 | 依赖 |
|------|-------|---------|------|
| 1 | ㉑ 测试配置 | SKILL.md 步骤 0 | 无 |
| 2 | ⑱ 临时文件 | SKILL.md 步骤 1 | 无 |
| 3 | ⑲ 跨论文污染 | SKILL.md 步骤 2 | 无 |
| ... | ... | ... | ... |

### 2.2 逐条呈现修复方案

对每个 issue，**单独呈现**修复方案给用户确认：

```markdown
### Issue ⑳：概念笔记"相关"区块缺少关联说明

**根因**：`obsidian-note-template.md` 模板中 `## 相关` 区块未要求关系说明，且位置在 `## 🔗 延伸探索` 之后

**修复方案**：
1. 模板 `## 相关` 每个链接追加 `——<一句话关系说明>`
2. `## 相关` 移至 `## ⚖️ 分析` 与 `## 🔗 延伸探索` 之间
3. 示例文件 `examples/concepts/self-attention.md` 同步更新

**涉及文件**：`references/obsidian-note-template.md`、`examples/concepts/self-attention.md`
**Tier**：A（单文件参考文档更新）
**必测模块**：T5（笔记生成）+ T8（产物结构）

是否同意此方案？
```

### 2.3 等待全部确认

- **逐条确认**：每个 issue 的方案单独向用户确认，不等全部列完
- **全部确认后**：进入 Phase 3（根因分析 + Advisor），然后 Phase 4（集中修改）
- **用户否决的方案**：回 Phase 1 调整后重新确认
- **用户要求修改的方案**：当场调整后再次确认

### Phase 2 出口

- [ ] 修复顺序已排定（若多 issue）
- [ ] **每个 issue 的方案已逐条获得用户确认**
- [ ] 所有确认的方案已汇总，准备集中修改
- [ ] **尚未修改任何代码**

## Phase 3: 根因分析 + Advisor 审查

### 3.1 诊断步骤

1. **复现**：跑 TESTING.md 中对应模块（参考 Phase 4 的映射表）
2. **对照 failure-modes.md**：查症状表
3. **若无匹配**：执行 failure-modes.md 附录中对应阶段的检测命令
4. **隔离**：缩小到具体文件/段落
5. **记录**：更新 ISSUES.md 条目的 `**问题分析**` 部分

### 3.2 参考文件依赖地图

| 症状区域 | 首要参考 | 次要参考 |
|----------|---------|---------|
| 配置行为 | SKILL.md 步骤 0 | frontmatter-schema.md（config 节） |
| 图片提取 | SKILL.md 步骤 1 | failure-modes.md F1-1 ~ F1-5 |
| 课程生成 | course-design-guide.md | SKILL.md 步骤 2 |
| 笔记结构 | obsidian-note-template.md | frontmatter-schema.md |
| 概念筛选 | SKILL.md 步骤 2 | obsidian-note-template.md（概念模板） |
| 问答行为 | qa-standards.md | SKILL.md（跨模块 Q&A） |
| 待学习机制 | SKILL.md（跨模块 to-learn） | frontmatter-schema.md（to-learn 节） |
| 命名/路径 | SKILL.md 命名约定 | failure-modes.md F5-1 |
| 跨论文功能 | SKILL.md（跨模块 cross-paper） | frontmatter-schema.md |

### 3.3 分派 Project Advisor 审查

**在动手修改前，必须分派 Advisor agent 进行审查。** 根据修复范围和场景选择审查深度：

| 修复范围 | 匹配场景 | Advisor 关注重点 |
|----------|---------|-----------------|
| 单文件小改（Tier A） | ⚡ 日常决策 | D1 必要性、I1 简洁性、M3 文档完整性 |
| SKILL.md 流程改动（Tier B） | 📐 功能设计 | D1-D4 方向 + A1-A2 架构 + M1-M4 维护 |
| 多文件管线变更（Tier C） | 🏗️ 架构变更 | 全部 16-20 维度 |

**分派方式**：使用 Agent 工具 spawn，prompt 模板：

> `对以下修复方案进行审查。场景：<匹配场景>。方案：<Phase 2 确认的方案摘要>。涉及文件：<文件列表>。请按 Project Advisor 框架输出审查报告。`

Advisor 返回审查报告后，若有 🔴 驳回或 🟠 简化建议，调整方案后重新走 Phase 2 确认。

### Phase 3 出口

- [ ] 根因已定位到具体文件 + 段落
- [ ] Advisor 审查报告已出，无 🔴 驳回项
- [ ] 若有方案调整 → 已回 Phase 2 重新获得用户确认

---

## Phase 4: 集中修复

> ⚠️ **全部方案确认 + Advisor 审查通过后，一次性修改所有文件。不逐条改。**

### 4.1 识别受影响文件

汇总所有确认方案的涉及文件列表，去重。原则：**不只修有 bug 的文件，检查所有交叉引用的参考文件是否有连锁影响。**

### 4.2 修复复杂度分级

| Tier | 范围 | 示例 |
|------|------|------|
| **A** | 单文件参考文档更新 | 修改 course-design-guide.md 的一条约束 |
| **B** | SKILL.md + 参考文档联动 | 修改流程指令 + 更新对应设计指南 |
| **C** | 多文件管线变更 | 改 SKILL.md + 参考文档 + 测试模块 + 示例文件 |

### 4.3 选择必测模块

对照 TESTING.md [快速索引：改动 → 必测模块](../test/TESTING.md#快速索引改动--必测模块)：

| 改动范围 | 必测模块 |
|----------|----------|
| 仅改 README/文案 | 无需测试 |
| 改 SKILL.md 流程步骤或顺序 | T9 全流程 + 对应步骤的子模块 |
| 改课程模板/生成规则 | T4 课程生成 |
| 改笔记模板/生成规则 | T5 笔记生成 |
| 改概念筛选标准 | T6 概念筛选与骨架 |
| 改示例嵌入规则 | T7 示例嵌入 |
| 改命名约定/产物目录 | T8 产物结构 |
| 改图片提取管线 | T3 图片提取 |
| 改问答标准 | T11 问答 |
| 改步骤 5 引导逻辑 | T10 步骤 5 引导 |
| 改待学习机制 | T13 待学习功能 |
| 改首次运行引导 | T1 环境检查 |
| 改跨论文去重或联系 | T12 跨论文功能 |
| 多处同时改动 | 各对应模块 + T9 全流程 |

### 4.4 集中修改

按 Phase 2 排定的顺序，逐文件修改。**每个文件的修改合并所有相关 issue 的改动，一次改到位。**

### 4.5 实现检查清单

- [ ] 修改主文件（按顺序，合并同文件改动）
- [ ] 更新交叉引用的参考文件
- [ ] 若新增约束/规则 → 更新对应测试模块的判定步骤
- [ ] 若填补了 failure-modes.md 的空白 → 添加/更新对应 failure mode 条目
- [ ] 若 README 有实质修改 → 记录变更日志条目（开发者寄语留占位 `___`）
- [ ] SKILL.md 行数不超过 500

### Phase 4 出口

- [ ] **所有确认方案的修改已集中完成**
- [ ] 必测模块已选定并汇总
- [ ] 变更日志条目已记录

---

## Phase 5: 测试执行

> ⚠️ **修复完成后统一跑测试。不跳过、不分开跑。**

### 5.1 执行测试

1. 运行 Phase 4 选定的所有必测模块（汇总去重）
2. 若有 Tier C 修复或触及步骤 0-5 核心逻辑 → 加跑 T9（全流程衔接）
3. 若路径/命名变更 → 加跑 T8（产物结构）
4. 若测试失败：分析是修复引入的回归还是测试期望需要更新

### 5.2 人工检查项

以下项目 agent 无法独立判定，需标记 `[人工]`：

- 图片内容正确性（T3）
- 示例相关性/贴合度（T7）
- 低分自测题（T4-C）
- 任何需要视觉判断的输出质量

### 5.3 分派 Advisor 做上线前审查

**提交前分派质量门神 advisor**，场景 🚀 上线前：

> `对以下改动进行上线前审查。变更摘要：<git diff --stat>。涉及 issue：<列表>。请按 Project Advisor 框架输出审查报告，重点：P1 测试覆盖、I3 边界处理、P2 部署风险。`

### 5.4 提交准备（硬门禁）

**在提交前必须完成以下步骤，不得跳过：**

1. 列出完整 changed files：`git status` 或 `git diff --stat`
2. 展示测试执行报告（按 TESTING.md 格式：汇总表 + 失败详情 + 待人工项）
3. 展示 Advisor 上线前审查报告
4. 草拟 commit message（风格匹配现有：`vX.Y · <简述>` 或 `<type>: <简述>`）
5. **将以上四项展示给用户，等待明确确认**
6. 用户确认后方可 commit

> ⚠️ 此门禁来自项目约定（memory: commit-requires-confirmation），无论用户是否在旁都必须遵守。

### Phase 5 出口

- [ ] 所有必测模块通过
- [ ] Advisor 上线前审查无 🔴 驳回
- [ ] 人工检查项已标记并提交用户
- [ ] 用户已确认 commit

## Phase 6: 文档 · 关闭 · 防回归

### 6.1 更新 ISSUES.md

每个解决后的 issue 条目下追加：

```markdown
> ✅ **已解决 (YYYY-MM-DD)**：<做了什么、改了哪些文件、设计决策。关联 failure-mode：F*-*>  
> **验证**：<测试结果摘要>
```

更新 ISSUES.md 头部计数（如 `当前 0 个待解决问题`）。

### 6.2 更新 failure-modes.md（条件性）

**提升标准**：Issue 满足以下全部条件时，应提升为 failure-modes.md 条目：

1. **可复现模式**：不是一次性错误，换了论文/触发条件仍可能发生
2. **可检测症状**：有不依赖完整上下文就能检查的症状
3. **标准修复流程**：有可编码的修复步骤，未来 agent 无需上下文即可执行

**不提升的情况**：一次性配置错误、网络瞬断、用户特定环境问题。

**提升时**：
- 按 `现象 → 原因（按概率排序）→ 修复 → 检测命令` 格式
- 按管线阶段放入对应章节（F0 ~ F5）
- 更新快速定位表（Section 1）
- 链接回 ISSUES.md 条目号

### 6.3 更新 README 变更日志

- 若修复值得版本升档 → 在变更日志表添加一行
- 批量修复 → 聚合为一行（如 v2.5 "6 项问题集中修复"）
- 开发者寄语留 `"___"` 占位，提醒用户填写

### 6.4 回归预防

- [ ] 每条新约束/规则 → 已有测试模块覆盖？未覆盖则新增判定步骤
- [ ] failure-modes.md 已覆盖此症状模式
- [ ] 若新增/扩展了测试模块 → 更新 TESTING.md 的「快速索引：改动 → 必测模块」表

### Phase 6 出口

- [ ] ISSUES.md 所有 issue 已关闭，头部计数已更新
- [ ] failure-modes.md 已更新（快速定位表 + 新条目 + 交叉引用）
- [ ] 测试覆盖无盲区
